SECTION 20 — RESEARCH GAP

The Concept Drift Problem — Why Static Models Fail

Real-world networks are not static datasets. Attackers constantly evolve their tactics. New vulnerabilities emerge. Traffic patterns shift. A model trained on January's attack signatures will gradually degrade on June's traffic. This is called concept drift — and most IDS research ignores it entirely.

What is concept drift?

Concept drift occurs when the statistical properties of the target variable change over time. In intrusion detection, this manifests as:

  • New attack variants: Attackers modify exploit techniques to evade detection (e.g., changing packet sizes, timing patterns, or protocol sequences).
  • Infrastructure changes: Network upgrades, new services, or configuration changes alter normal traffic patterns.
  • Seasonal variations: Business cycles cause shifts in legitimate traffic volume and characteristics.
  • Zero-day exploits: Entirely new attack classes that didn't exist during training.
⚠️ The static model assumption: Traditional ML-based IDS assume that training data and deployment data come from the same distribution. This assumption breaks down immediately in production. A model achieving 91% accuracy on test data can drop to 74% within weeks of deployment as attack patterns evolve.

Why existing solutions fall short

Most published IDS research focuses on maximizing accuracy on fixed benchmark datasets. Common limitations include:

❌ No Adaptation Mechanism

Models are trained once and deployed indefinitely. No monitoring for distribution shift. No retraining triggers.

❌ Unrealistic Evaluation

Train/test splits are random samples from the same time period. Doesn't simulate temporal drift.

❌ Manual Intervention Required

When performance degrades, human operators must identify the issue and manually retrain the model.

❌ False Positive Explosion

As drift occurs, models become overly sensitive, flagging legitimate traffic as attacks.

The Adaptive Gatekeeper approach

Our solution draws inspiration from recent research in adaptive machine learning for network security. The key insight: don't wait for catastrophic performance collapse — detect drift proactively and adapt automatically.

We implement a three-component system:

  1. ADWIN Drift Detector: Continuously monitors prediction accuracy using adaptive windowing to detect statistically significant changes.
  2. Kolmogorov-Smirnov Test: Validates detected drift by comparing feature distributions between old and new data windows.
  3. Shadow Model Retraining: When drift is confirmed, trains a new model on recent data in parallel, validates it, then swaps it into production.
💡 Research contribution: While drift detection exists in generic ML literature, applying it to multi-class network intrusion detection with real-time constraints and class imbalance is an open challenge. This implementation demonstrates a practical, deployable solution.

Component 1: ADWIN (Adaptive Windowing)

ADWIN is a change detection algorithm designed for data streams. It maintains a sliding window of recent prediction outcomes and automatically adjusts the window size based on observed variance.

How ADWIN works

  • Sliding window: Keeps recent predictions (e.g., last 1,000 samples) in memory.
  • Sub-window comparison: Splits the window into two parts and compares their statistical properties.
  • Drift signal: If the mean accuracy between sub-windows differs significantly (using Hoeffding bounds), ADWIN raises a drift alert.
  • Adaptive size: Window shrinks during stable periods (to save memory) and expands during uncertain periods (to gather more evidence).
from river.drift import ADWIN

# Initialize detector
drift_detector = ADWIN(delta=0.002)

# For each prediction
for actual, predicted in stream_predictions():
    is_correct = (actual == predicted)
    drift_detector.update(is_correct)
    
    if drift_detector.drift_detected:
        print("⚠️ Drift detected at flow", flow_count)
        trigger_drift_validation()

Why ADWIN?

  • Memory-efficient: Doesn't store all historical data, just a compressed window.
  • Parameter-free: Only one hyperparameter (δ = confidence level), not sensitive to tuning.
  • Theoretical guarantees: Provides statistical bounds on false positive rate.
  • Fast: O(log n) update time per sample.

Component 2: Kolmogorov-Smirnov Test

ADWIN detects changes in accuracy. But what if accuracy drops due to random noise rather than true concept drift? The KS-test validates drift by examining whether the input feature distributions have changed.

How KS-test works

  • Two-sample test: Compares the distribution of a feature in the "old" window vs "new" window.
  • Null hypothesis: Both samples come from the same distribution.
  • Test statistic: Maximum distance between the two cumulative distribution functions (CDFs).
  • Decision rule: If p-value < 0.05 for multiple critical features, reject null hypothesis → drift confirmed.
from scipy.stats import ks_2samp

# Compare feature distributions
old_window = feature_buffer[:1000]
new_window = feature_buffer[1000:2000]

drift_features = []
for feature in critical_features:
    stat, p_value = ks_2samp(old_window[feature], new_window[feature])
    if p_value < 0.05:
        drift_features.append(feature)

if len(drift_features) >= 3:
    print(f"✓ Drift confirmed in {len(drift_features)} features")
    trigger_model_retraining()

Why KS-test?

  • Prevents false alarms: ADWIN alone might trigger on random accuracy fluctuations. KS-test confirms the underlying data has actually changed.
  • Non-parametric: Makes no assumptions about feature distributions (works with skewed, multi-modal data).
  • Interpretable: Tells you which features drifted, aiding root cause analysis.

Component 3: Shadow Model Retraining

When drift is confirmed, we can't just immediately replace the production model — the new model might perform worse! Shadow modeling tests the new model safely before deployment.

Shadow model workflow

  1. Trigger: ADWIN + KS-test both confirm drift.
  2. Data collection: Gather last N samples (e.g., 10,000 flows) from the stream buffer.
  3. Train shadow model: Retrain LightGBM on recent data using same hyperparameters as original model.
  4. Parallel evaluation: For the next M flows, make predictions with both old and new models (but only serve the old model's predictions to users).
  5. Performance comparison: If shadow model accuracy > old model accuracy + ε (e.g., +2%), promote shadow to production.
  6. Rollback safety: If shadow model performs worse, discard it and wait for more data.
def train_shadow_model(recent_data, recent_labels):
    # Clone production model config
    shadow_model = clone_model(production_model)
    
    # Retrain on recent data
    shadow_model.fit(recent_data, recent_labels)
    
    # Evaluate on validation window
    val_acc = shadow_model.score(validation_X, validation_y)
    prod_acc = production_model.score(validation_X, validation_y)
    
    if val_acc > prod_acc + 0.02:
        promote_to_production(shadow_model)
        log(f"✓ Model updated: {prod_acc:.3f} → {val_acc:.3f}")
    else:
        log(f"✗ Shadow model rejected: {val_acc:.3f} ≤ {prod_acc:.3f}")

Experiment design

To simulate realistic concept drift, we partition the UNSW-NB15 dataset chronologically (not randomly):

Dataset Partitioning

Part 1 (Training): First 60% of data (sorted by timestamp) — represents "normal" attack patterns from Period A.

Part 2 (Drift Simulation): Last 40% of data — represents evolved attack patterns from Period B.

Drift injection point: Flow #52,000 (first major distribution shift detected).

Evaluation metrics

  • Accuracy before drift: Model performance on Part 1 test set (no drift).
  • Accuracy during drift: Model performance immediately after drift introduction (without adaptation).
  • Accuracy after adaptation: Model performance after shadow model promotion.
  • Drift detection latency: Number of samples between drift start and detection.
  • False positive rate: Number of false drift alerts in stable regions.
SECTION 22 — IMPLEMENTATION

Implementation Details

Translating the methodology into production-ready Python code. This section covers the actual implementation, edge cases, and engineering decisions.

Data structures and buffering

The drift detection system operates on streaming data, requiring efficient circular buffers to store recent samples without unbounded memory growth.

import numpy as np
from collections import deque

class DriftMonitor:
    def __init__(self, buffer_size=10000, adwin_delta=0.002):
        self.buffer_size = buffer_size
        self.feature_buffer = deque(maxlen=buffer_size)
        self.label_buffer = deque(maxlen=buffer_size)
        self.prediction_buffer = deque(maxlen=buffer_size)
        
        # ADWIN detector for accuracy monitoring
        from river.drift import ADWIN
        self.drift_detector = ADWIN(delta=adwin_delta)
        
        # Tracking metrics
        self.drift_points = []
        self.flow_count = 0
        self.retraining_history = []
    
    def update(self, features, true_label, predicted_label):
        """Process one new sample"""
        self.flow_count += 1
        
        # Add to buffers
        self.feature_buffer.append(features)
        self.label_buffer.append(true_label)
        self.prediction_buffer.append(predicted_label)
        
        # Update ADWIN with correctness signal
        is_correct = int(true_label == predicted_label)
        self.drift_detector.update(is_correct)
        
        # Check for drift
        if self.drift_detector.drift_detected:
            print(f"⚠️  ADWIN drift alert at flow {self.flow_count}")
            self.validate_drift()
    
    def validate_drift(self):
        """Run KS-test on feature distributions"""
        if len(self.feature_buffer) < 2000:
            return  # Not enough data
        
        # Split buffer into old/new windows
        mid = len(self.feature_buffer) // 2
        old_features = np.array(list(self.feature_buffer)[0:mid])
        new_features = np.array(list(self.feature_buffer)[mid:])
        
        # Run KS-test on critical features
        from scipy.stats import ks_2samp
        critical_features = [0, 1, 2, 5, 8, 10]  # Top features by SHAP
        
        drifted_features = []
        for feat_idx in critical_features:
            stat, p_value = ks_2samp(
                old_features[:, feat_idx],
                new_features[:, feat_idx]
            )
            if p_value < 0.05:
                drifted_features.append(feat_idx)
        
        # Confirm drift if 3+ features changed
        if len(drifted_features) >= 3:
            print(f"✓ Drift confirmed ({len(drifted_features)} features)")
            self.drift_points.append(self.flow_count)
            self.retrain_shadow_model()
        else:
            print(f"✗ False alarm (only {len(drifted_features)} features)")

Shadow model training

The retraining process must be non-blocking (doesn't interrupt live predictions) and must validate the new model before deployment.

    def retrain_shadow_model(self):
        """Train new model on recent data"""
        # Extract recent samples
        recent_X = np.array(list(self.feature_buffer)[-5000:])
        recent_y = np.array(list(self.label_buffer)[-5000:])
        
        # Split into train/validation
        split_idx = int(len(recent_X) * 0.8)
        train_X, val_X = recent_X[:split_idx], recent_X[split_idx:]
        train_y, val_y = recent_y[:split_idx], recent_y[split_idx:]
        
        # Train shadow model (clone of production model)
        from lightgbm import LGBMClassifier
        shadow_model = LGBMClassifier(
            n_estimators=100,
            learning_rate=0.1,
            max_depth=7,
            random_state=42
        )
        shadow_model.fit(train_X, train_y)
        
        # Compare performance
        shadow_acc = shadow_model.score(val_X, val_y)
        prod_acc = self.production_model.score(val_X, val_y)
        
        print(f"Shadow: {shadow_acc:.3f} | Production: {prod_acc:.3f}")
        
        # Promote if better
        if shadow_acc > prod_acc + 0.02:  # 2% improvement threshold
            self.production_model = shadow_model
            self.retraining_history.append({
                'flow': self.flow_count,
                'old_acc': prod_acc,
                'new_acc': shadow_acc
            })
            print("✓ Model updated successfully")
            return True
        else:
            print("✗ Shadow model rejected (no improvement)")
            return False

Integration with production system

The drift monitor runs alongside the FastAPI backend, logging predictions and triggering retraining without blocking the main inference pipeline.

# In main.py
from fastapi import FastAPI, BackgroundTasks
import threading

app = FastAPI()
drift_monitor = DriftMonitor(buffer_size=10000)
model_lock = threading.Lock()

@app.post("/predict")
async def predict(data: FlowData, background: BackgroundTasks):
    # Make prediction
    with model_lock:
        features = preprocess(data)
        prediction = drift_monitor.production_model.predict(features)[0]
    
    # Update drift monitor (async, doesn't block response)
    background.add_task(
        drift_monitor.update,
        features,
        data.true_label,  # Assuming ground truth available
        prediction
    )
    
    return {"prediction": prediction}

Handling edge cases

Insufficient Data

If buffer has <2000 samples, skip KS-test (not statistically reliable). Wait for more data before validating drift.

Repeated False Alarms

If ADWIN triggers >3 times in 1000 flows without KS confirmation, increase ADWIN delta (reduce sensitivity).

Catastrophic Drift

If accuracy drops >20% suddenly, bypass shadow validation and immediately retrain (emergency mode).

Model Reversion

Keep last 3 production models in memory. If new model fails validation, can roll back to previous stable version.

⚠️ Limitations and future work:
